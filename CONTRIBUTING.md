# Contributing

This is a personal scaffold repository. Contributions are welcome but accepted selectively — I maintain this to fit my own workflow, not as a general-purpose framework.

## What's likely to be accepted

### In the kernel (tier 1) — `01-kernel/`

Content here is meant to be universal. Good contributions:

- **Corrections to principles** — if a claim in `PHILOSOPHY.md` or `principles/*.md` is empirically wrong or the citation is flawed, fix it.
- **New patterns in `patterns/`** — if you've discovered a truly general pattern that fits the scaffold's philosophy.
- **Template improvements** — cleaner slot syntax, better defaults in `templates/`.
- **Script hardening** — bug fixes, cross-platform improvements to hooks and install scripts.
- **New universal meta-skills or meta-agents** — that would benefit any agentic workflow, not just mine.

### In the stack (tier 2) — `02-stack/`

Content here is opinionated. Good contributions:

- **Bug fixes** in install scripts, config samples
- **Adding a layer / tool alternative** with clear documentation of why it fits the stack's shape
- **Pattern pages** (cross-device, parallelization, etc.) with broader technique coverage

Not accepted: swapping out my picks. Tool picks are personal; fork the stack repo and maintain your own.

### In the docs site (`site/`)

- Rendering / UX fixes
- Better accessibility
- Component improvements (PageTitle badges, etc.)

## What's NOT likely to be accepted

### In the work tier (tier 3) — `03-work/`

This is my personal instance. Contributions here don't make sense — it's a worked example, not a template. Fork and build your own work tier.

### Tool recommendations that change my picks

If you're going to suggest I use Neovim instead of VS Code, or OpenCode instead of Claude Code — fine suggestions for your setup, but not for mine. Open an issue if you want to discuss, don't submit a PR.

### Expansions that bloat the scaffold

This scaffold optimizes for **progressive disclosure**. Adding 10 new skills because they might be useful isn't a win — it's a cost. Before adding anything, consider whether it needs to be in the scaffold or whether it lives better in a separate project.

## Contribution process

1. **Open an issue first** for anything non-trivial. Saves time.
2. **Fork the repo** (once the kernel is extracted to its own public repo, the kernel fork is the place for kernel PRs).
3. **Branch** off `main`.
4. **Match the style** — short commits, descriptive titles, no Claude-as-coauthor tags, no `🤖 Generated with...` footers.
5. **Include frontmatter** — `title`, `description`, `stratum`, `status`, `date`, `tags` as appropriate.
6. **Stratum-correct placement** — kernel content must genuinely be universal; don't push stratum-5 content into the kernel.
7. **PR description** — explain what and why. Link to the issue.
8. **Be patient** — this is a personal repo; I review when I have time.

## Code & commit style

- **No AI attribution in commits.** No "Co-Authored-By: Claude" lines, no "Generated by Claude Code" footers, no vendor robot-emoji footers. If you used AI assistance, that's fine — but the commit attribution is yours, not the AI's. This applies equally to every AI tool (Claude, Copilot, Cursor, Aider, etc.). A global `commit-msg` hook enforces this across every repo on this machine — see [`research/personal-workflow/no-ai-attribution-git-hooks.md`](./research/personal-workflow/no-ai-attribution-git-hooks.md) for the policy rationale and full setup (hook script, `core.hooksPath`, template dir, husky/lefthook per-repo fixes, and an optional GitHub Actions server-side check).
- **Short, descriptive commit titles.** "Fix CRLF handling in stratum frontmatter injector" > "Fixed a bug".
- **Conventional Commits is OK but not required.** Clear > ceremonial.
- **Line endings:** LF (Unix) for files in this repo. The `.gitattributes` (when added) will enforce.
- **No emojis in code** unless explicitly needed.
- **Respect the scaffold's principles** — progressive disclosure, convention-as-compressed-decision, etc. See [`01-kernel/principles/`](./01-kernel/principles/).

## Reporting issues

- **Bugs** — include repro steps, expected vs actual, environment (OS, tool versions).
- **Doc inaccuracies** — link to the specific file + line number.
- **Security concerns** — do NOT open a public issue for anything security-sensitive. Email me directly or use GitHub's private security advisory feature.

## Kernel repo (future)

Once [`01-kernel/`](./01-kernel/) is extracted to its own repo, the kernel becomes a proper open-source project with its own contribution culture. Until then, contributions to the monorepo are rare and careful.
